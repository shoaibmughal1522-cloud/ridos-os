Create Your GitHub Account

GitHub is where your RIDOS OS code lives, gets built automatically, and is shared with the world. It's free.

✓ Already have an account? Skip to the account settings below — there are a few things to configure before creating the repo.
🌐
1. Sign up at github.com
Go to github.com/signup. Choose a username carefully — it will appear in your project URL as github.com/YOUR-USERNAME/ridos-os. Pick something professional.
✉️
2. Verify your email
GitHub will send a verification email. You cannot create public repositories or enable GitHub Actions until your email is verified.
🔑
3. Enable Two-Factor Authentication (2FA)
Go to Settings → Password and authentication → Two-factor authentication. Use an authenticator app (Google Authenticator, Authy). This is critical for an open-source project — your account will be the only one with write access.
🖼
4. Set your public profile
Go to Settings → Public profile. Add your name and a short bio. This is what contributors and users see when they visit the repository.
  • Required Account created and email verified
  • Required 2FA enabled
  • Optional Profile photo and bio added

Install & Configure Git

Run these commands once on your Linux build machine before you can push code.

▸ Run these on your Linux build machine The machine where you have the ridos-v1.1 folder — your Ubuntu/Debian system.
bash — Install Git
# Install Git (if not already installed) sudo apt update && sudo apt install -y git # Verify installation git --version # Expected: git version 2.x.x
bash — Configure your identity (do this once)
# Replace with YOUR name and email git config --global user.name "Your Name" git config --global user.email "[email protected]" # Set default branch name to 'main' git config --global init.defaultBranch main # Better diff output git config --global core.pager "less -FX" git config --global color.ui auto # Verify git config --list
bash — Generate SSH key for GitHub authentication
# Generate Ed25519 key (modern, secure) ssh-keygen -t ed25519 -C "[email protected]" -f ~/.ssh/github_ridos # Start ssh-agent and add key eval "$(ssh-agent -s)" ssh-add ~/.ssh/github_ridos # Print your PUBLIC key — copy this to GitHub cat ~/.ssh/github_ridos.pub
🔑
Add SSH key to GitHub
Copy the output of cat ~/.ssh/github_ridos.pub → go to GitHub → Settings → SSH and GPG keys → New SSH key → Title: "RIDOS Build Machine" → Paste key → Save. Then test: ssh -T [email protected]
bash — Test GitHub SSH connection
ssh -T [email protected] # Expected: Hi YOUR-USERNAME! You've successfully authenticated...
  • Required Git installed and configured (name + email)
  • Required SSH key added to GitHub account
  • Required ssh -T [email protected] returns success

Stage & Create Your First Commit

A commit is a snapshot of your project at a point in time. The first commit establishes the foundation.

▸ Your project structure before committing Make sure your ridos-v1.1 folder looks like this:
ridos-v1.1/ ├── .gitignore ← tells Git what to ignore ├── README.md ← project homepage on GitHub ├── CHANGELOG.md ← version history ├── SECURITY.md ← vulnerability reporting ├── legal/ │ ├── LICENSE.txt ← GPL v3 │ ├── COPYRIGHT ← your copyright notice │ └── CONTRIBUTORS.md ← contributors list ├── ridos-core/ │ ├── ai_daemon.py │ ├── ridos_shell.py │ ├── intent_parser.py │ ├── web_search.py │ └── setup_wizard.py ├── build-system/ │ ├── scripts/lb_config.sh │ └── config/ │ ├── package-lists/ridos.list.chroot │ ├── hooks/0100-ridos-setup.hook.chroot │ └── includes.binary/boot/grub/... └── .github/ ├── workflows/build-iso.yml ├── ISSUE_TEMPLATE/bug_report.md ├── ISSUE_TEMPLATE/feature_request.md └── PULL_REQUEST_TEMPLATE.md
bash — Initialize and stage all files
# Navigate to your project folder cd ~/ridos-v1.1 # Initialize Git repository git init -b main # Check what files will be committed git status # Stage ALL files for the first commit git add . # Double-check what's staged git status # Every file should show in green under "Changes to be committed"
⚠ Before committing — verify no secrets are staged Run git status and make sure you do NOT see any api.key, .env, or *.pem files. The .gitignore should block these, but always double-check.
bash — Create the first commit

Push to GitHub

Connect your local repository to GitHub and push the first commit.

bash — Connect local repo to GitHub
# Replace YOUR-USERNAME with your GitHub username git remote add origin [email protected]:YOUR-USERNAME/ridos-os.git # Verify the remote was added correctly git remote -v # Should show: # origin [email protected]:YOUR-USERNAME/ridos-os.git (fetch) # origin [email protected]:YOUR-USERNAME/ridos-os.git (push) # Push your first commit git push -u origin main
✅
What happens after the push
1. Your code appears on github.com/YOUR-USERNAME/ridos-os
2. GitHub Actions automatically detects .github/workflows/build-iso.yml
3. The ISO build starts within seconds
4. You can watch the build at github.com/YOUR-USERNAME/ridos-os/actions
bash — Tag your first release
# Create a version tag — this triggers a GitHub Release git tag -a v1.1.0 -m "RIDOS OS v1.1.0 (Basra) — Initial Release" # Push the tag to GitHub git push origin v1.1.0 # This triggers the release workflow in build-iso.yml # GitHub will create a Release with the ISO attached for download
✓ Your project is now live! Anyone can find it at github.com/YOUR-USERNAME/ridos-os and see your full codebase, license, and documentation.
  • Required Remote added and push succeeds
  • Required Code visible on GitHub
  • Optional v1.1.0 tag pushed and Release created

GitHub Actions — Automated ISO Builds

Every push to main automatically builds a fresh RIDOS OS ISO in the cloud. No Linux machine needed.

🏗
How the CI/CD pipeline works
git push → GitHub detects push → starts Ubuntu 22.04 runner → installs live-build → runs lb build → uploads ISO as artifact → optionally creates a GitHub Release.
TriggerWhat happensWhen to use
git push origin mainBuilds ISO, uploads as artifact (30-day expiry)Every code change
git push origin v1.x.xBuilds ISO + creates public GitHub ReleaseOfficial releases only
Manual triggerBuild ISO on demand from GitHub UITesting without a code change
📦
Download the built ISO
Go to github.com/YOUR-USERNAME/ridos-os/actions → click the latest build → scroll down to Artifacts → download ridos-os-1.1.0-iso.zip. Extract it to get the .iso file.
bash — Trigger a manual build from the CLI
# Install GitHub CLI (gh) sudo apt install gh # Login gh auth login # Trigger the build workflow manually gh workflow run build-iso.yml # Watch the build logs live gh run watch
✓ Free CI/CD minutes Public repositories get unlimited free GitHub Actions minutes. Building a full Linux ISO takes about 30–60 minutes per run.

Branch Strategy

How to organize your work so the main branch always has working code.

BranchPurposeRules
main Always production-ready. Every commit here triggers a build. Protected — only merge via PR
develop Integration branch. New features merge here first. Merge into main when stable
feature/name One feature or fix per branch. Delete after merging
hotfix/name Urgent fixes that go directly to main. Merge to both main and develop
bash — Daily workflow example
# Start a new feature git checkout -b feature/xfce-branding # Make your changes... # Then commit git add . git commit -m "feat(desktop): add RIDOS XFCE theme and wallpaper" # Push the feature branch git push origin feature/xfce-branding # On GitHub: open a Pull Request from feature/xfce-branding → main # Review, merge, delete the feature branch # Back to main and pull latest git checkout main git pull origin main
bash — Protect the main branch (run once on GitHub)
# Using GitHub CLI gh api repos/YOUR-USERNAME/ridos-os/branches/main/protection \ --method PUT \ --field required_status_checks='{"strict":true,"contexts":[]}' \ --field enforce_admins=false \ --field required_pull_request_reviews='{"required_approving_review_count":0}' \ --field restrictions=null # Or go to: Settings → Branches → Add branch protection rule # Branch name pattern: main # ✓ Require a pull request before merging
✓ You're all set! RIDOS OS is on GitHub, building automatically, and protected against accidental breaking changes. Share the link and accept contributions!
  • Recommended develop branch created
  • Recommended main branch protection enabled
  • Once First feature branch workflow tested

Commit Message Reference

Professional commit messages make your project history readable and enable automated changelogs.

▸ Format: Conventional Commits type(scope): short description

This is the standard used by Angular, Vue, and most major open-source projects.
TypeWhen to useExample
featNew feature addedfeat(shell): add /network command
fixBug fixfix(daemon): handle timeout on slow connections
buildBuild system changesbuild: add grub theme to live-build config
docsDocumentation onlydocs: update README with build instructions
refactorCode restructure, no behavior changerefactor(parser): simplify intent detection
securitySecurity fixsecurity: restrict api.key file permissions
choreMaintenance taskschore: update .gitignore patterns
ciCI/CD changesci: add artifact retention to build workflow
